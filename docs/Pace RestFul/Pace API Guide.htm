Introduction

Basic SOAP Details

SOAP is defined as Simple Object Access Protocol and is an XML-based protocol to exchange structured information. You can use SOAP over variety of networking protocols. SOAP is a W3C Recommendation.

Tip: If you are new to SOAP, or want to review it, an internet search on “SOAP” plus your desired programming language can provide ample resources for informational purposes.

The following lists the basic details and building blocks of SOAP:

  • SOAP Envelope

    <soap:Envelope xmlns:soap="http://www.w3.org/2001/12/soap-envelope"
      soap:encodingStyle="http://www.w3.org/2001/12/soap-encoding">

    --------------------------------------------------------------

    </soap:Envelope>

  • Header

    The header is optional. It contains the application specific information (Generic information).

    <soap:Header>

    <m:Trans xmlns:m="http://www.w3schools.com/transaction/" soap:mustUnderstand="1">

    234

    </m:Trans>

    </soap:Header>

  • Body

    The body contains call and response information and is a mandatory element in SOAP. This element includes the actual SOAP message intended for the endpoint.

    <soap:Envelope xmlns:soap="http://www.w3.org/2001/12/soap-envelope"
      soap:encodingStyle="http://www.w3.org/2001/12/soap-encoding">

    <soap:Body>

    <m:GetPrice xmlns:m="http://www.w3schools.com/prices/">

    <m:GetPrice xmlns:m="http://www.w3schools.com/prices/">

    <m:Item>Apples</m:Item>

    </soap:Body>

    </soap:Header>

  • Fault Element

    The fault element contains error and status information.

Basic WSDL Details

WSDL (Web Services Description Language) is an XML-based language for defining the web services. WSDL enables two applications to share data based on standard integration rules for transferring data. WSDL is a W3C Recommendation.

Tip: If you are new to WSDL, or want to review it, an internet search on “WSDL” plus your desired programming language can provide ample resources for informational purposes.

The major elements of WSDL are:

  • Definitions

    A definition element is the root element which defines the name of the web service and declares the namespaces.

    <definitions xmlns="http://schemas.xmlsoap.org/wsdl/" xmlns:ns1="http://rpc.services.appbox.pace2020.com"
      xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/" xmlns:soapenc="http://schemas.xmlsoap.org/soap/encoding/"
      xmlns:tns="urn://pace2020.com/epace/sdk/AttachmentService" xmlns:wsdl="http://schemas.xmlsoap.org/wsdl/"
      xmlns:wsdlsoap="http://schemas.xmlsoap.org/wsdl/soap/" xmlns:xsd="http://www.w3.org/2001/XMLSchema"
      targetNamespace="urn://pace2020.com/epace/sdk/AttachmentService">

    --------------------------------------------------------------

    </definitions>

  • Types

    A type describes the data types used.

    <wsdl:types>

    <xsd:schema targetNamespace="urn://pace2020.com/epace/sdk/CreateObject" elementFormDefault="qualified"
      attributeFormDefault="qualified">

    <xsd:element name="createActivityCode">

    <xsd:complexType>

    <xsd:sequence>

    <xsd:element name="activityCode" type="ns1:activityCode" minOccurs="1" maxOccurs="1"/>

    </xsd:sequence>

    </xsd:complexType>

    </xsd:element>

    --------------------------------------------------------------

    </wsdl:types>

  • Message

    A message communicates data and describes a one way message either through request or response.

    <message name="createQuoteCategoryResponse">

    <part element="tns:createQuoteCategoryResponse" name="parameters"/>

    </message>

  • Port Type

    A port type is a set of operations supported by one or more endpoints.

    <portType name="CreateObjectPortType">

    <operation name="createWorkstationActivityCode">

    <input message="tns:createWorkstationActivityCodeRequest" name="createWorkstationActivityCodeRequest"/>

    <output message="tns:createWorkstationActivityCodeResponse" name="createWorkstationActivityCodeResponse"/>

    </operation>

    --------------------------------------------------------------

    </portType>

  • Binding

    Binding is a protocol and data format specification for a particular port type.

    <binding name="CreateObjectHttpBinding" type="tns:CreateObjectPortType">

    <wsdlsoap:binding style="document" transport="http://schemas.xmlsoap.org/soap/http"/>

    <operation name="createWorkstationActivityCode">

    <wsdlsoap:operation soapAction=""/>

    <input name="createWorkstationActivityCodeRequest">

    <wsdlsoap:body use="literal"/>

    </input>

    <output name="createWorkstationActivityCodeResponse">

    <wsdlsoap:body use="literal"/>

    </output>

    </operation>

    </binding>

  • Service

    Service is the location / address for invoking a specified service.

    <service name="CreateObject">

    <port binding="tns:CreateObjectHttpBinding" name="CreateObjectHttpPort">

    <wsdlsoap:address location="http://viperfish/rpc/company:public/services/CreateObject"/>

    </port>

    </service>

Midmarket Suite API Service

The Midmarket Suite API (Application Program Interface) service enables other systems outside of Midmarket Suite to interact with Midmarket Suite through an industry-standard series of SOAP-based services. It enables two main types of interactions:

  • Generic Object Access

    Through the Midmarket Suite API, other systems can obtain generic access to objects and its fields including UDO’s and UDF’s. It provides full CRUD (Create, Read, Update and Delete) capabilities against any object in Pace. The following are generic object accesses:

    • Create – Add a new object
    • Read - Read attributes of an existing object
    • Update – change attributes of an existing object
    • Delete – Delete an existing object
    • Clone – Duplicate an existing object
    • Find – Search for a specific object
    • Attachments – Read, update or delete attachments for a specific objects

  • Specific Process Execution

    • Midmarket Suite API can invoke specific processes such as process checks, post checks, generate auto bills, generate auto invoicing, post bill batches, post GL batches, post invoices, process job material prepare pick tickets, process job material commit pick tickets, process WIP, compute customer service charges, calculate the current accounting period, close the current accounting period, close the fiscal year, reopen the last accounting period, post inventory, process job cost totals reset, invoke the scheduler EDI, process job transactions, reset the customer end of year, renumber bill checks, auto pay bills, auto invoice fulfillments jobs, reconcile commissions, refresh job plans and execute PaceConnect records within Midmarket Suite.

    Tip: The majority of your work with Midmarket Suite is likely through CRUD services (Create, Read, Update and Delete). The majority of Midmarket Suite’s business rules are implemented as events that react upon these core operations.

To assist your developers in programming the API, the Remote API customization tool includes access to the Software Development Kit (SDK) on the Admin Console. This SDK contains code samples in Java, .net, PHP.

Set up the Midmarket Suite Remote API

Note: You can perform the tasks in the API section of this guide only if you purchased the Remote Application Program Interface (API) customization tool. For more information or a quote, please contact a member of our National Sales team at 800-875-7117.

Verify Your API License

  • From the main menu, navigate to Administration, System Setup, General, System Information. The System Information page appears.
  • On the System Information page, click on the Customization Licenses tab.
  • In the License column of the Customization group box, locate the API module and verify that Yes appears in the Licensed field

    • Note: Contact your Midmarket Suite Support representative if the Licensed field is disabled for the API module.

Set up a user with API Permissions

Note: This discrete permission feature inside Midmarket Suite governs access to the API. The standard security system applies to the browser user interface only.

Important: Object and Attribute permissions do NOT apply when you use the API.

  • To set up a specific security group to use the Remote API module, from the main menu, navigate to Administration, System Setup, General, System Groups. The System Groups List page appears.
  • On the System Groups List page, do one of the following:
    • Click [Add New Record] to add a new security group. The System Group Add page appears.
    • Click the magnifying glass icon beside the existing group for which you want to grant API permissions. The System Group Detail page appears.
  • If you are on the System Group Add page, in the Group Name field (required) enter a descriptive name for this security group and click [Add]. The System Group Detail page appears, with the Object Added message at the top of the page.
  • On the Group Detail page, on the context navigational menu at the top of the page, click Permissions. The Permissions Detail page appears.
  • On the Permissions Detail page, on the Permissions tab, check the box in the Allow Remote API Usage field.
  • Click [Update]. The Permissions Detail page refreshes, with the Updated message at the top of the page.
  • Add or locate the User record for which you want assign API permissions.
  • Assign the desired user to the user group that includes Remote API access permissions.

The Create API Service

Use this API to create and persist data object to the Midmarket Suite system. WSDL URL: http://your.host.com/rpc/services/CreateObject?wsdl

To create a new instance of an object:

  • Construct an instance of the object locally.
  • Fill in any fields for which you want to supply values.
  • Pass the instance to the appropriate create method.

If the Create process was successful:

  • It returns a new object instance that corresponds to the newly-created object.
  • The object is persisted in the database.
  • The system fills any fields with default values.

The Read API Service

Use this API to read any data object from the Midmarket Suite system. WSDL URL: http://your.host.com/rpc/services/ReadObject?wsdl

  • Fupply the unique identifier.
  • Pass the instance to the appropriate read method.

The Find Object API Service

The Find Object service of the Remote API enables you to search Midmarket Suite for data objects that match a set of criteria you supply. The criterion is in the form of an XPath expression. WSDL URL: http://your.host.com/rpc/services/FindObjects?wsdl

The Find method requires the following two parameters of type=String:

  • The object type, such as Job, for which you want to search.
  • An XPath expression, such as “@job='XYZ',” for how to filter the objects.

Tip: The XPath expression can reference other objects. For example, if you wanted to search for all job objects for a given customer, you can use an expression of “../@id = 'HOUSE'.” This method returns a list of unique identifiers.

For performance reasons, you can then use the Read method to load object instances, using the key values. You control how many object instances the system reads and at what frequency.

The following are some example calls for the Find method:

  • find( "Job", "../@id = 'HOUSE'" )
  • find( "Job", "@poNum != 'test me'" )
  • find( "Employee", "@department = '001'" )
  • find( "JobTransaction", "activityCode/@chargeBasis = 6 and @job='XYZ' and @jobPart='01'" )

In additional to the Find method, the find object service also enables you to sort on the keys retrieved and is based on the XPath criterion defined on XPathDataSort.

Refer to the following example calls for the findAndSort method:

  • findAndSort( "Job", "../@id = 'HOUSE'", sort )
  • findAndSort( "Job", "@poNum != 'test me'", sort )
  • findAndSort( "Employee", "@department = '001'", sort )

where “sort” is an array of XPathDataSort on which you want to set two parameters:

  • Parameter1: A boolean that determines if the sort order is ascending or descending.
  • Parameter2: The XPath criterion on which you want to perform the sort.

    • Example:

    • xPathDataSort.descending = true;
    • xPathDataSort.xpath = "@description";

    • data object - The data object on which the attachment is set.

    • attribute - The attribute name that is an attachment field.
    • attachment - The attachment instance created.
  • To retrieve all attachments associated with a data object or its attribute, call the method getAllAttachments by passing the below parameters. This returns an array of the attachments.

    • primary key - The primary key of the data object.
    • attribute - The attribute name that is an attachment field.

    • attribute - The attribute name that is an attachment field.

Initial Web Service Setup in Java

In the Midmarket Suite SDK, the Static Client Programming Model (Dynamic proxy client) implements examples where the artifacts are created on the client side based on the WSDL information. Do all of the following using Apache ANT build tool:

  • Download the WSDL files from your host by providing the user name and password for the user created with API permissions.
  • Generate the Service Endpoint Interface (SEI) to access the published web service.
  • Execute the target “compile” and run required target for running the samples.

The following are sample codes to accomplish the above steps:

  • Download WSDL files from the host to the client side.

    • <!-- Target to download wsdl -->

    <target name="download-wsdl">

    <get username="${epace.username}" password="${epace.password}" src=http://${epace.host}:${epace.port}/rpc/services/${service.name}?wsdl dest="${wsdl.dir}/${service.name}.wsdl"/>

    </target>

    <!-- Target that invokes download-wsdl to download a required WSDL -->

    <target name="get-wsdl" description="retrieve wsdl files from epace">

    <mkdir dir="${wsdl.dir}"/>

    <antcall target="download-wsdl">

    <param name="service.name" value="Version" />

    </antcall>

    ----------------------------------

    </target>

  • Generate Java classes from WSDL information on the client side to get access to the published web services from the Midmarket Suite System. wsimport tool is used to parse published WSDL and generate JAX-WS artifacts on the client side to access the web service. External JAXB binding file “Bindings.xml” is mentioned in the target.

    • <!-- Generic target to generate Java classes from WSDL -->

    <target name="wsdl2java">

    <wsimport destdir="${build.dest}" sourcedestdir="${gensrc.dir}" wsdl="${basedir}/build/wsdl/${name}.wsdl" wsdllocation=" http://${epace.host}:${epace.port}/rpc/services/${name}?wsdl">

    <binding dir="${basedir}/etc" includes="Bindings.xml"/>

    </wsimport>

    </target>

    <!-- Target invoking wsdl2java for generating Java Source -->

    <target name="gen-src" depends="get-wsdl" description="generate java source from wsdl files">

    <mkdir dir="${gensrc.dir}"/>

    <antcall target="wsdl2java">

    <param name="name" value="CreateObject" />

    </antcall>

    ----------------------------------

    </target>

Pre-authenticate web service request in C#

Override the generated stub to add GetWebRequest(Uri uri) method to preauthenticate.

  • Create HttpWebRequest instance from Uri.
  • Create NetworkCredential instance by “Basic” authentication technique.
  • If NetworkCredential instance is not null add "Basic " + Convert.ToBase64String(credentialBuffer); to the request header.
  • If NetworkCredential instance is null throw Exception.
  • Return the HttpWebRequest instance.

    protected override System.Net.WebRequest GetWebRequest(Uri uri)
    {
     HttpWebRequest request;
     request = (HttpWebRequest)base.GetWebRequest(uri);
     NetworkCredential networkCredentials = Credentials.GetCredential(uri, "Basic");
     if (networkCredentials != null)
     {
      byte[] credentialBuffer = new UTF8Encoding().GetBytes(networkCredentials.UserName + ":" + networkCredentials.Password);
      request.Headers["Authorization"] = "Basic " + Convert.ToBase64String(credentialBuffer);
     }
     else
     {
      throw new ApplicationException("No network credentials");
     }
     return request;
    }
